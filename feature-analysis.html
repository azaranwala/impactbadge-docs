1. Executive Summary

ImpactBadge is a comprehensive K-12 volunteer hour tracking platform spanning 4 applications with 158+ backend Cloud Functions, 75+ mobile features, and 10 web portal pages. The platform serves 4 user roles (Student, Org Admin, School Admin, Parent) with full gamification, compliance reporting, and college export capabilities.

2. Platform Overview

2.1 iOS App (Swift/SwiftUI)

  • Architecture: MVVM with @Observable ViewModels
  • Min Target: iOS 16+
  • Extensions: Apple Watch app, Home Screen Widget
  • Auth: Email/Password, Google, Apple Sign-In, Clever SSO
  • Navigation: 6-tab layout (Explore, Hours, Profile, Leaderboard, Messages, Alerts)

2.2 Android App (React Native/TypeScript)

  • Architecture: React Navigation with hooks-based state management
  • Auth: Email/Password, Google, Clever SSO
  • Navigation: Role-based tabs (Student: 5 tabs, Org Admin: 4 tabs, Parent: 3 tabs)
  • Localization: i18next (English, Spanish)

2.3 Web Portal (React/TypeScript/Vite)

  • Framework: React 18 + TailwindCSS 4.0 + Recharts
  • Purpose: Admin/School dashboards, Clever SSO login
  • Auth: Clever SSO, Firebase Auth
  • Deployment: Firebase Hosting (impacthub-prod.web.app)

2.4 Backend (Firebase Cloud Functions)

  • Runtime: Node.js 20, TypeScript (strict mode)
  • Functions: 158 callable/HTTP/trigger functions
  • Validation: Zod schema validation on all inputs
  • Geo: geofire-common for proximity searches

5. End-to-End Scenarios

Scenario 1: Student Registration & First Event


1. Student opens app → sees onboarding tutorial (4 slides)
2. Taps "Create Account" → enters email, password, name, grade, school
3. System auto-detects school from email domain
4. If under 13: parent email required → COPPA consent email sent
5. Student selects volunteer interests (environment, education, health, etc.)
6. Lands on Home tab → sees nearby events via GPS
7. Browses events by category or searches via natural language
8. Taps event card → views full details, address, capacity
9. Registers for event → receives confirmation
10. Gets push notification reminder 24 hours before event

Status: TESTED (AuthTests, EventsTests)

Scenario 2: Event Check-In & Hours Logging


1. Student arrives at event location
2. Opens check-in screen → GPS validates within 300m
3. Alternative: enters 4-digit PIN or scans QR code
4. Check-in confirmed → timer starts
5. After volunteering, student checks out
6. Hours automatically calculated and logged
7. Hours appear as "pending" in dashboard
8. Org admin sees pending verification in admin queue
9. Admin approves hours → student notified
10. Hours move to "verified" → XP awarded → badge check triggered

Status: TESTED (CheckInTests, verifyHours.test.ts)

Scenario 3: Self-Reported Hours


1. Student volunteers at non-ImpactBadge event
2. Goes to Hours tab → taps "Self-Report Hours"
3. Enters: organization, date, hours, description
4. Draws e-signature on signature pad
5. Optionally attaches photo evidence
6. Submits → hours appear as "pending self-report"
7. Org admin reviews evidence (signature + photos)
8. Admin approves/rejects → student notified
9. If approved, hours added to verified total

Status: TESTED (SelfReportScreen.test.tsx, submitSelfReportedHours)

Scenario 4: Gamification Progression


1. Student completes first event → earns "First Step" badge
2. XP points awarded based on hours + category bonuses
3. Streak counter increments for consecutive days
4. Student advances levels: Greenhorn → Volunteer → Helper → Hero → Champion → Legend → Sage
5. Student checks leaderboard (global, school, friends)
6. Joins time-limited challenge ("Log 10 hours this week")
7. Completes challenge → claims XP reward
8. Generates referral code → shares with friends
9. Friend signs up with code → both earn bonus XP

Status: TESTED (leaderboard.test.ts, challenges.test.ts, referrals.test.ts, awardBadges.test.ts)

Scenario 5: Parent Monitoring Flow


1. Student generates 6-digit link code from profile
2. Parent installs app → creates parent account
3. Parent enters link code → accounts linked
4. Parent sees child dashboard: hours, events, badges, streak
5. Parent configures permissions (block categories, limit events/week)
6. Parent receives notifications for: check-ins, badge earned, hours verified
7. Parent can unlink child account at any time

Status: TESTED (generateLinkCode.test.ts, getChildDashboard.test.ts, parentPermissions.test.ts)

Scenario 6: Org Admin Event Management


1. Org admin logs in → lands on admin dashboard
2. Creates new event: title, description, date, location, capacity, category
3. System generates check-in PIN and QR code
4. Optionally creates shifts for multi-slot events
5. Students register → admin sees registration count
6. During event: admin monitors check-ins in real-time
7. After event: admin reviews pending hour verifications
8. Admin bulk-approves verified hours
9. Views organization analytics: total events, volunteers, hours
10. Posts announcement to all registered volunteers

Status: TESTED (createEvent.test.ts, AdminTests, getOrgAnalytics.test.ts, bulkVerifyHours.test.ts)

Scenario 7: School Admin Compliance


1. School admin logs in via Clever SSO
2. Views school dashboard: total students, hours, compliance %
3. Sets graduation hour requirements (e.g., 75 hours)
4. Monitors per-student compliance status
5. Reviews HCPS pre-approval requests
6. Bulk imports student roster from CSV
7. Generates compliance report (PDF/CSV)
8. Exports to WinSchool or Naviance
9. Views district-level aggregate reporting

Status: TESTED (getSchoolStudents.test.ts, getSchoolHoursSummary.test.ts, exportSchoolHoursCSV.test.ts)

Scenario 8: College Application Export


1. Student nears graduation → opens College Export
2. Selects date range for hours summary
3. Generates PDF certificate with verified hours
4. Exports Common App-compatible activity record
5. Generates Naviance CSV for school counselor
6. Creates full volunteer portfolio (hours, reflections, skills, badges)
7. Downloads/shares documents

Status: TESTED (generatePortfolioCall.test.ts, pdfGenerator.test.ts, certificateFields.test.ts)

Scenario 9: Messaging & Team Collaboration


1. Student finds another volunteer on leaderboard
2. Sends follow request → views public profile
3. Opens direct message conversation
4. Creates team for upcoming event → gets join code
5. Shares code with friends → friends join team
6. Team registers for event together
7. Uses group chat to coordinate logistics

Status: TESTED (sendMessage.test.ts, createTeam.test.ts, joinTeam.test.ts, followStudent.test.ts)

Scenario 10: Apple Watch Quick Check-In


1. Student arrives at event wearing Apple Watch
2. Watch shows active event from synced data
3. Taps "Check In" → GPS validates location
4. Alternative: enters PIN on watch
5. Haptic feedback confirms check-in
6. Hours sync back to iPhone app

Status: NOT TESTED (Watch app has no unit tests)

10.2 Composite Indexes (23)

Key indexes for performance:

  • events: geoHash + startDate (geo-proximity queries)
  • events: category + startDate (category filtering)
  • users: role + totalHours (leaderboards)
  • users: role + schoolName + totalHours (school rankings)
  • users: role + interests (ARRAY_CONTAINS for matching)
  • conversations: participants + lastMessageAt (chat list)
  • notifications: uid + createdAt (activity feed)
  • selfReportedHours: studentUid + status (admin review)
