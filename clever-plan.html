4. Phase 1: District SSO (Instant Login)

What Changes

With Clever Complete, ImpactBadge can add a "Log in with Clever" button directly on the login page (Instant Login). Currently, Library users must launch from the Clever portal.

Already Done

  • OAuth flow is complete — same flow works for both Library and Complete
  • The "Sign in with Clever" button already exists on the web portal
  • Android has a Clever button (feature-flagged)
  • iOS has Clever deep link handling

What's New

1a. Enable Instant Login in Clever Dashboard

  • Go to Clever Dashboard → Settings → Instant Login
  • Set redirect URI: https://impacthub-prod.web.app/auth/clever/callback
  • Enable all supported user types (student, teacher, staff, district_admin)
  • Clever provides an Instant Login URL: https://clever.com/oauth/authorize?response_type=code&redirect_uri={URI}&client_id={ID}

1b. Update iOS Login Screen

The iOS "Sign in with Clever" button is currently hidden behind a feature flag. For Complete districts:

  • Enable enable_clever_sso in Firebase Remote Config
  • The OAuth URL already uses the correct Clever authorize endpoint
  • No code changes needed — just enable the flag

1c. Shared Device Session Override

Clever requires that a new SSO request overrides any existing session:

  • Current behavior: cleverOAuthCallback.ts creates a custom token for the Clever user. If another user was signed in, Firebase Auth will sign them out on the client when signInWithCustomToken is called. This already satisfies the requirement.
  • Verification needed: Confirm that the iOS/Android clients properly sign out the previous user before signing in with the Clever token.

5. Phase 2: Secure Sync (Rostering)

Architecture Overview


District SIS (PowerSchool, Infinite Campus, etc.)
    ↓ (nightly sync)
Clever Platform (normalizes + cleanses data)
    ↓ (district app token)
ImpactBadge Backend (Cloud Function)
    ↓ (writes to Firestore)
Firestore: users, schools, sections collections
    ↓
iOS / Android / Web apps

Two Token Types

TokenPurposeHow to Get
SSO Bearer TokenIdentify a single user during loginOAuth code exchange (already built)
District App TokenAccess ALL data for a connected districtClever Dashboard → District tokens

The SSO token is per-user (already working). The district app token is new — it lets ImpactBadge query all students, teachers, sections, and schools for a connected district without any user being logged in.

  syncedTeachers: 180,
  syncedSections: 450,
  connectedAt: Timestamp,

Security: District app tokens are highly sensitive — they grant access to all student PII. Store encrypted in Firestore with Cloud Functions-only access (deny client reads in firestore.rules).

2b. Roster Sync Cloud Function

NEW: cleverRosterSync — Scheduled Pub/Sub function (runs daily at 2:00 AM EST)


// Pseudo-code for the sync flow
async function syncDistrict(districtId: string, appToken: string) {
  // 1. Fetch all schools
  const schools = await cleverAPI.get("/v3.1/schools", appToken);
  for (const school of schools) {
    await upsertSchool(school);
  }

  // 2. Fetch all students
  const students = await cleverAPI.get("/v3.1/users?role=student", appToken);
  for (const student of students) {
    // Match existing user or create new
    await upsertStudent(student);
  }

  // 3. Fetch all teachers
  const teachers = await cleverAPI.get("/v3.1/users?role=teacher", appToken);
  for (const teacher of teachers) {
    await upsertTeacher(teacher);
  }

  // 4. Fetch all sections (classes)
  const sections = await cleverAPI.get("/v3.1/sections", appToken);
  for (const section of sections) {
    await upsertSection(section);
  }

  // 5. Archive users no longer in Clever data
  await archiveRemovedUsers(districtId, syncedIds);
}

2c. User Upsert Logic (Student)


async function upsertStudent(cleverStudent) {
  // Priority matching (same as SSO, but using district token data)
  // 1. Match by cleverUserId
  // 2. Match by studentId + schoolName
  // 3. Match by email
  // 4. Pre-provision in pendingStudents (don't create Firebase Auth — user hasn't logged in yet)

  const updateData = {};
  // Only set non-empty values (same merge logic as cleverOAuthCallback)
  if (cleverStudent.name.first) updateData.displayName = `${cleverStudent.name.first} ${cleverStudent.name.last}`;
  if (cleverStudent.email) updateData.cleverEmail = cleverStudent.email;
  if (cleverStudent.student_number) updateData.studentId = cleverStudent.student_number;
  if (cleverStudent.grade) updateData.grade = parseInt(cleverStudent.grade);
  if (cleverStudent.school) updateData.schoolName = cleverStudent.school.name;
  // ... demographics (v3.1)

  updateData.cleverUserId = cleverStudent.id;
  updateData.cleverDataType = "complete";
  updateData.cleverSyncedAt = FieldValue.serverTimestamp();
}

2d. Archive/Restore Logic

When a student is removed from Clever (e.g., transferred to another district):

  • Archive: Set cleverArchived: true, cleverArchivedAt: timestamp
  • Do NOT delete the user or their hours — historical data must be preserved
  • Remove from compliance reports and school student lists

When a student reappears in Clever:

  • Restore: Set cleverArchived: false, update all Clever fields
  • Restore in compliance reports and school lists
  • Historical hours/badges/XP preserved

2e. Events API (Roster Changes)

Instead of full daily sync for changes, poll Clever Events API:

  • Endpoint: GET /v3.1/events?school={id}&record_type=users
  • Event types: users.created, users.updated, users.deleted, sections.created, sections.updated, sections.deleted
  • Poll every 15 minutes or on-demand
  • Process incrementally — much faster than full sync

Implementation: Scheduled Cloud Function cleverEventsPoll (every 15 min)

2f. Section/Class Data

Clever sections map to ImpactBadge's school context:

  • Used by teachers to see "their" students (future: scoped by section instead of whole school)
  • Used for school-vs-school challenges (section-level competition)
  • Stored in cleverSections/{sectionId} collection

6. Phase 3: LMS Connect (Grade Passback)

What It Does

Allows ImpactBadge to:

  1. LTI 1.3 Launch: Students/teachers launch ImpactBadge from their LMS (Canvas, Schoology, etc.)
  2. Grade Passback: Push volunteer hours as "grades" or "completion" back to the LMS gradebook
  3. Course Mapping: Map ImpactBadge volunteer activities to LMS courses

Prerequisites

  • Must be existing Secure Sync partner (Phase 2 complete)
  • Requires Clever API v3.1
  • Requires Clever Complete agreement

Use Case for ImpactBadge

  • Teacher assigns "Complete 10 volunteer hours this semester" as an LMS assignment
  • Student completes hours in ImpactBadge
  • ImpactBadge pushes completion status + hours count back to the LMS gradebook
  • Teacher sees volunteer hours alongside academic grades

Design Decisions (Before Building)

  1. When to push assignments: After teacher creates a challenge/requirement in ImpactBadge
  2. When to push results: After hours are verified (auto or manual)
  3. LMS import strategy: Push-only (ImpactBadge → LMS), no bidirectional sync

Recommendation: DEFER TO v2.0

LMS Connect is valuable but complex (5+ days of work). The SSO + Rostering integration provides immediate value for HCPS. LMS Connect can be added after certification for SSO + Secure Sync is complete.
